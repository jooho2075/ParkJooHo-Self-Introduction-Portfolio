### BACK-END
**🔹 미니 프로젝트:** 우리카드 결제 MSA (wooricard-payment-msa)

가맹점(POS)과 카드사로부터 받은 결제 승인 요청을 중계하고, 표준 전문 규격 데이터를 처리해 결제 이력을 관리하는 **VAN(Value Added Network) 게이트웨이**를 담당했습니다. POS → VAN → 카드사 → 은행으로 이어지는 결제 승인 흐름을 마이크로서비스로 구현한 5인 팀 프로젝트의 일부입니다.

| 항목 | 내용 |
|------|------|
| 기술 스택 | Java, Spring Boot, OpenFeign, Eureka, MySQL |
| 담당 파트 | VAN 게이트웨이 — POS 접점 API 설계, 가맹점→카드사 데이터 중계 로직, 결제 이력 DB 관리 |
| 팀 구성 | 5인 (MSA 인프라, 서버 분석/Swagger, POS·VAN 로직, **VAN 서버**, 은행 서비스) |
| 링크 | [GitHub](https://github.com/yujung23/wooricard-payment-msa.git) |

**시스템 흐름 (POS → VAN → 카드사)**
1. POS로부터 `PosPaymentRequest` 수신
2. 카드번호 기반으로 카드사(VISA/MASTERCARD/WOORICARD) 식별 후 카드사 요청 객체로 변환
3. OpenFeign을 통해 카드사 서버로 승인 요청 전달
4. 카드사 응답을 `PaymentHistory` 엔티티로 DB 저장
5. 최종 승인 결과를 `PosPaymentResponse`(Record)로 POS에 반환

**핵심 기술 포인트 — Java Record로 DTO 설계**
- 카드번호·유효기간·거래금액 등 결제 정보는 생성 후 변경되면 안 되는 데이터라, Class 대신 **Record**를 사용해 모든 필드를 `private final`로 선언 → 프로세스 도중 값 변조로 인한 데이터 오염을 원천 차단
- Boilerplate 코드를 제거해 `PosPaymentRequest`/`PosPaymentResponse`가 '읽기 전용' 객체임을 코드 구조 자체로 명확히 전달

**비즈니스 로직**
- 카드번호 패턴으로 카드사(VISA/MASTERCARD/WOORICARD) 식별
- 카드사 승인 결과 수신 + DB 기록을 `@Transactional`로 하나로 묶어 데이터 유실 방지
- 주문 번호(POS)·승인 번호(STAN) 기반 로그로 결제 흐름 추적성 확보

**테스트 코드** (`@SpringBootTest` + `@MockitoBean`으로 카드사 서버 Mock 처리)
| 테스트 케이스 | 검증 내용 |
|------|------|
| VISA 카드 정상 승인 | 응답 데이터 매핑 및 DB 저장 정상 수행 확인 |
| 잔액 부족(51) | 카드사 응답 코드·메시지가 POS로 정상 전달되는지 확인 |
| 미정의 카드번호 | UNKNOWN 카드사로 정상 식별되는지 확인 |

**🔹 미니 프로젝트:** 우리카드 소비 트렌드 분석 대시보드
- 카드 상품 기획·마케팅 담당자를 위한 내부 대시보드로, 약 530만 건의 소비 데이터를 8개 카테고리로 집계해 분기별 트렌드를 시각화했습니다. WAS 이중화·DB 복제·세션 클러스터링을 고려한 3-Tier Architecture로 구현했습니다.

| 항목 | 내용 |
|------|------|
| 기술 스택 | Java 17, Servlet/JSP, Tomcat, MySQL 8(Replication), Nginx, Docker |
| 아키텍처 | 3-Tier (Presentation-Application-Data), Tomcat 2대 + Nginx 라운드로빈, MySQL Source/Replica |
| 담당 역할 | **MySQL Replication Read/Write 분리** — 조회(Read)는 Replica, 쓰기(Write)는 Source로 역할 분리, GTID 기반 자동 위치 동기화 구성 |
| 핵심 문제 해결 | ① 530만 건 실시간 집계로 인한 응답 지연 → 사전 집계 테이블 도입<br>② WAS 단일 장애점(SPOF) → Tomcat 이중화 + Nginx 라운드로빈<br>③ **DB Read/Write 부하 분산** → MySQL Source/Replica 복제 구성 (담당)<br>④ 이중화 환경 세션 정합성 → DeltaManager 세션 클러스터링 |
| 링크 | [GitHub](https://github.com/light11014/woori-card-project.git) |

**🔹 백엔드 기술 세미나:** 주제: Redis
- RDBMS로 대용량 데이터를 처리할 때 발생하는 Disk I/O 병목·쿼리 부하 문제를 짚고, 이를 해결하는 Redis의 구조와 RDBMS와의 협업 전략을 발표 자료로 직접 준비하고 발표했습니다.

| 항목 | 내용 |
|------|------|
| 다룬 내용 | Redis가 필요한 이유, In-Memory Key-Value 구조와 장단점, Redis&RDBMS 협업 전략(Look Aside / Write Around / Write Back) |
| 코드 시연 | 5만 명 동시접속 티켓 예매 상황을 RDBMS(H2, File Mode)와 Redis(Docker, In-Memory, Sorted Set)로 각각 구현해 처리 시간 비교 |
| 시연 결과 | 동일 로직 기준 RDBMS 112.01s vs Redis 108.69s — RAM과 SSD의 물리적 읽기/쓰기 속도 차이를 직접 입증 |
| 결론 | 특정 DB에 의존하기보다 저장 매체의 물리적 특성을 고려한 아키텍처 설계가 중요 |
